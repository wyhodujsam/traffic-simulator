# Plan 11-07: Sorted Vehicle List for O(log n) Leader Lookup

## Goal

Replace the O(n) linear scan in `Lane.getLeader()` / `findLeaderAt()` / `findFollowerAt()` with O(log n) binary search on a sorted list. The list is sorted descending by position and re-sorted once per tick after physics. This fixes the performance bottleneck with 200+ vehicles.

## Context

Currently Lane uses linear scans for leader/follower lookup (3 methods, each O(n)). With n vehicles per lane and m lanes, each physics tick is O(n^2) per lane. The PRD recommends Option B: maintain a sorted ArrayList, sort once per tick after physics (O(n log n) per lane instead of O(n^2)).

After Plan 11-01, the `vehicles` list is private and accessed only through Lane methods. This makes it safe to change the internal data structure.

### Task 1: Implement sorted vehicle list in Lane

<read_first>
- `backend/src/main/java/com/trafficsimulator/model/Lane.java`
</read_first>

<action>
Modify Lane.java:

1. **Change addVehicle()** to insert in sorted position (descending by position):
   ```java
   public void addVehicle(Vehicle v) {
       // Insert maintaining descending position order
       int insertIdx = 0;
       for (int i = 0; i < vehicles.size(); i++) {
           if (vehicles.get(i).getPosition() > v.getPosition()) {
               insertIdx = i + 1;
           } else {
               break;
           }
       }
       vehicles.add(insertIdx, v);
   }
   ```

2. **Add resortVehicles()** — called once per tick after physics updates positions:
   ```java
   public void resortVehicles() {
       vehicles.sort(Comparator.comparingDouble(Vehicle::getPosition).reversed());
   }
   ```

3. **Rewrite getLeader()** using binary search on sorted list (descending = index 0 is frontmost):
   ```java
   public Vehicle getLeader(Vehicle vehicle) {
       // List is sorted descending. Leader = first vehicle with position > vehicle.position
       // which means it's the vehicle just before `vehicle` in the list (lower index)
       int idx = -1;
       for (int i = 0; i < vehicles.size(); i++) {
           if (vehicles.get(i) == vehicle) { idx = i; break; }
       }
       if (idx <= 0) return null;  // vehicle is at front or not found
       return vehicles.get(idx - 1);
   }
   ```

4. **Rewrite findLeaderAt()** using binary search:
   ```java
   public Vehicle findLeaderAt(double position) {
       // Sorted descending. Find first vehicle with position > given position.
       // Binary search for the insertion point.
       Vehicle closest = null;
       for (int i = vehicles.size() - 1; i >= 0; i--) {
           Vehicle v = vehicles.get(i);
           if (v.getPosition() > position) {
               closest = v;
               break;
           }
       }
       return closest;
   }
   ```

5. **Rewrite findFollowerAt()** similarly — find nearest vehicle behind position.

Note: For the initial implementation use linear scan on the sorted list (which is still faster because we can break early). A full binary search optimization can be added later if profiling shows need.
</action>

### Task 2: Add resortVehicles() call to PhysicsEngine

<read_first>

<action>
At the end of `PhysicsEngine.tick(Lane lane, double dt, double stopLinePosition)`, after all vehicles have been updated, add:

```java
// Re-sort after position updates to maintain sorted invariant
lane.resortVehicles();
```

Also remove the in-method sort (line 46) since the list is now maintained sorted:
```java
// REMOVE: vehicles.sort(Comparator.comparingDouble(Vehicle::getPosition).reversed());
```

The PhysicsEngine processes vehicles front-to-back. Since the list is already sorted descending, no initial sort is needed — just iterate in order. The resort at the end restores the invariant after position changes.

Note: PhysicsEngine.tick() now operates on a copy (`new ArrayList<>(lane.getVehiclesView())`) per Plan 11-01, so the in-method sort was on the local copy anyway. The `lane.resortVehicles()` at the end sorts the internal list.
</action>
