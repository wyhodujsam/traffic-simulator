# Plan 11-08: ReentrantReadWriteLock Synchronization

## Goal

Fix the race condition (bug C1 from review) where the tick thread and STOMP thread can modify RoadNetwork simultaneously. Add a ReentrantReadWriteLock to SimulationEngine: tick pipeline holds readLock, command dispatch holds writeLock.

## Context

Currently TickEmitter runs on a `@Scheduled` thread at 20 Hz, iterating over RoadNetwork (roads, lanes, vehicles). Simultaneously, STOMP messages can trigger SimulationEngine.enqueue() followed by drainCommands() at tick start. The actual race is: tick thread reads vehicles while a command (e.g., CloseLane) modifies the same collection. After Plan 11-04, CommandDispatcher handles mutations — adding the writeLock there provides a single synchronization point.

After Plan 11-04, drainCommands() delegates to CommandDispatcher.dispatch(). The lock strategy:
- **readLock**: TickEmitter.emitTick() holds readLock for the entire pipeline (drain + physics + snapshot)
- **writeLock**: CommandDispatcher.dispatch() acquires writeLock for each command — but since dispatch is called from drainCommands() which is inside the readLock, we need a different approach.

**Revised strategy**: The drainCommands() call happens at tick start BEFORE acquiring readLock, or we use writeLock for drainCommands + readLock for the rest. Best approach: writeLock around drainCommands, readLock around physics-through-snapshot.

```java
private final ReentrantReadWriteLock networkLock = new ReentrantReadWriteLock();
```

Wrap `drainCommands()` in SimulationEngine with writeLock:

```java
public void drainCommands() {
    List<SimulationCommand> pending = new ArrayList<>();
    commandQueue.drainTo(pending);
    if (pending.isEmpty()) return;

    writeLock().lock();
    try {
        for (SimulationCommand cmd : pending) {
            commandDispatcher.dispatch(cmd);
        }
    } finally {
        writeLock().unlock();
    }
}
```

This ensures all command mutations happen under writeLock. No changes needed in CommandDispatcher itself — the lock is held by the caller.

### Task 3: Add readLock to TickEmitter pipeline

<read_first>
- `backend/src/main/java/com/trafficsimulator/scheduler/TickEmitter.java`
</read_first>

<action>
Restructure `emitTick()` to use locks:

```java
@Scheduled(fixedRate = 50)
public void emitTick() {
    long tickStart = System.nanoTime();

    // 1. Drain commands FIRST (acquires writeLock internally)
    simulationEngine.drainCommands();

    // 2. Read-only pipeline under readLock
    simulationEngine.readLock().lock();
    try {
        RoadNetwork network = simulationEngine.getRoadNetwork();

        long tick;
        if (simulationEngine.getStatus() == SimulationStatus.RUNNING) {
            tick = simulationEngine.getTickCounter().incrementAndGet();
            if (network != null) {
                // ... entire physics pipeline (spawn, physics, lane change, intersections, despawn)
            }
        } else {
            tick = simulationEngine.getTickCounter().get();
        }

        // Build snapshot under readLock (reads vehicle state)
        SimulationStateDto state = snapshotBuilder.buildSnapshot(...);
        statePublisher.broadcast(state);
    } finally {
        simulationEngine.readLock().unlock();
    }

    // Timing monitoring (outside lock)
    long elapsedMs = (System.nanoTime() - tickStart) / 1_000_000;
    // ... warn if slow
}
```

Note: tickCounter.incrementAndGet() is atomic and safe under readLock. The writeLock is only needed for structural mutations (adding/removing vehicles, changing status).

**Important**: spawn and despawn operations mutate collections (addVehicle/removeVehicle). Since these run inside the tick pipeline (not from external threads), they are safe under readLock because no other thread holds a conflicting lock at this point. The writeLock protects against STOMP-thread commands running concurrently with the tick.

Actually, this is incorrect — readLock allows multiple readers but blocks writers. If the tick pipeline mutates (spawn/despawn), it should use writeLock for the entire tick. Revised approach:

**Use writeLock for the entire tick pipeline** (since it mutates state), and readLock only for snapshot-building if we split them. But the simplest correct approach is writeLock for the full tick:

```java
simulationEngine.writeLock().lock();
try {
    // drain + tick pipeline + snapshot
} finally {
    simulationEngine.writeLock().unlock();
}
```

This is correct but means STOMP commands wait for tick completion. Since tick takes <40ms and commands are queued anyway, this is acceptable.

**Final approach**: Use writeLock for the entire emitTick() body. Commands from STOMP threads only call enqueue() (which uses ConcurrentLinkedQueue/BlockingQueue — already thread-safe). The writeLock prevents any external direct mutation during tick.
</action>

```java
@SpringBootTest
class ConcurrencySafetyTest {

    @Autowired SimulationEngine engine;
    @Autowired TickEmitter tickEmitter;

    /**
     * 100 threads send commands while tick is running.
     * No ConcurrentModificationException should occur.
     */
    @Test
    void concurrentCommandsDuringTick_noException() throws Exception {
        // Start simulation
        engine.enqueue(new SimulationCommand.Start());
        tickEmitter.emitTick();  // process start

        ExecutorService executor = Executors.newFixedThreadPool(10);
        CountDownLatch latch = new CountDownLatch(100);
        AtomicInteger errors = new AtomicInteger(0);

        // Spawn 100 command-sending threads
        for (int i = 0; i < 100; i++) {
            final int idx = i;
            executor.submit(() -> {
                try {
                    // Mix of different commands
                    if (idx % 3 == 0) {
                        engine.enqueue(new SimulationCommand.SetSpawnRate(1.0 + idx * 0.1));
                    } else if (idx % 3 == 1) {
                        engine.enqueue(new SimulationCommand.SetSpeedMultiplier(1.0));
                    } else {
                        engine.enqueue(new SimulationCommand.SetMaxSpeed(33.33));
                    }
                    // Simultaneously run a tick
                    if (idx % 5 == 0) {
                        tickEmitter.emitTick();
                    }
                } catch (Exception e) {
                    errors.incrementAndGet();
                } finally {
                    latch.countDown();
                }
            });
        }

        latch.await(10, TimeUnit.SECONDS);
        executor.shutdown();

        assertEquals(0, errors.get(), "Expected zero concurrency errors");
    }

    /**
     * Verify lock doesn't cause deadlock — tick completes within timeout.
     */
    @Test
    void tickCompletesUnderLoad_noDeadlock() throws Exception {
        engine.enqueue(new SimulationCommand.Start());
        tickEmitter.emitTick();

        // Run 50 ticks with concurrent commands
        ExecutorService executor = Executors.newFixedThreadPool(4);
        for (int i = 0; i < 50; i++) {
            executor.submit(() -> engine.enqueue(new SimulationCommand.SetSpawnRate(1.0)));
        }

        // All ticks must complete within 5 seconds
        long start = System.currentTimeMillis();
        for (int i = 0; i < 50; i++) {
            tickEmitter.emitTick();
        }
        long elapsed = System.currentTimeMillis() - start;

        executor.shutdown();
        assertTrue(elapsed < 5000, "50 ticks should complete within 5s, took " + elapsed + "ms");
    }
}
```
</action>
