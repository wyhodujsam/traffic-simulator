## Task 1 — Repro Evidence (AC#6)

### Method

Backend was placed into a guaranteed-503 state by moving the osm2streets binary aside:

```bash
cd /home/sebastian/apps/traffic-simulator
mv backend/bin/osm2streets-cli-linux-x64 backend/bin/osm2streets-cli-linux-x64.repro-bak
cd backend && mvn spring-boot:run    # NO overrides
```

Frontend was started (`cd frontend && npm run dev`), browser opened to http://localhost:5173/map, bbox positioned on the canonical Modlin area used throughout Phase 24.1 + Phase 24.2 (`{south=52.431, west=20.65, north=52.438, east=20.662}`), DevTools Network tab opened, "Fetch roads (osm2streets)" button clicked.

The resolver returned the configured path unchanged when the binary file did not exist (per Plan 24.2-01's Iron-Law: "the resolver returns the configured path UNCHANGED so callers see clean ENOENT — never silently fabricate a phantom path"). ProcessBuilder produced ENOENT. OsmController's `@ExceptionHandler` for `Osm2StreetsCliException` translated the failure to HTTP 503 with the canonical JSON body.

### Outcome: B — no bug

**HTTP status (Network tab):** `503`

**Response body (Network tab, verbatim):**
```json
{"error":"osm2streets unavailable. Please try again later."}
```

**Sidebar text (right pane of /map page, verbatim):**
```
osm2streets unavailable. Please try again later.
```

**Disposition:** The sidebar text matches the backend `error` field byte-for-byte. The `response.json().catch(() => ({error:'Unknown error'}))` fallback in `MapPage.handleFetchRoadsO2s` (lines 110-145) never fires because the response body IS valid JSON with an `error` field. `MapSidebar.ErrorContent` (line 184) renders the propagated `error` prop verbatim.

**Conclusion:** No frontend bug. The user's recollection of "Unknown error" was paraphrasing the actual displayed message. The CONTEXT.md AC#6 risk note ("possibly a no-bug — worth a 5-minute reproduction at the start of the phase") is confirmed correct.

### Cleanup

Binary was restored after the repro:

```bash
mv backend/bin/osm2streets-cli-linux-x64.repro-bak backend/bin/osm2streets-cli-linux-x64
```

Verified via `ls -la backend/bin/`:
```
osm2streets-cli-linux-x64
osm2streets-cli-linux-x64.sha256
README.md
```

(No `.repro-bak` file remains.)

## Decisions Made

- **AC#6 outcome B accepted without an additional regression-net Vitest.** The plan's optional clause permits adding one; we deferred. Rationale: the existing `osm-bbox-o2s.spec.ts` (Phase 24 e2e — stubbed 200/3-roads response) and `osm-bbox-real-backend.spec.ts` (Phase 24.1 e2e — real backend, real road count) cover the success-path sidebar contract end-to-end. The 503-failure-path contract is now empirically witnessed in this SUMMARY's "Task 1 — Repro Evidence" section. Adding a unit-level Vitest for `MapSidebar` rendering of an arbitrary `error` prop would duplicate coverage of code that the repro proved already correct.
- **AC#8 smoke run BEFORE the SUMMARY commit, not after.** The plan's Task 3 was a blocking checkpoint:human-verify gate ahead of the final docs commit. The user's resume signal (`approved: no-jvm-overrides: confirmed; phase-18: 18; phase-23: 20; phase-24: 10`) closes the gate; the SUMMARY then captures the evidence and the final commit closes the plan.
- **No code change in Plan 04 of any kind.** Outcome B precluded a frontend fix. AC#8 is a smoke audit, not an implementation task. The whole plan is documentation + evidence capture — the appropriate shape for an AC-closure plan that depends on Plans 01-03 having already shipped the foundational fix.
- **Endpoint and schema corrections applied silently to the human-facing checkpoint instructions.** Showing the user the wrong endpoint name (`/api/osm/fetch-roads-osm4j` vs the actual `/api/osm/fetch-roads-gh`) or the wrong bbox shape (`{min_lon, min_lat, ...}` vs `{south, west, north, east}`) would have produced a 404 / 422 and wasted the user's smoke time. Documented as "tech-stack patterns" in the frontmatter for future plan authors.
