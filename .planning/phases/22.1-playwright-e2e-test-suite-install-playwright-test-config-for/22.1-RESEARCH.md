## 1. `@playwright/test` installation and baseline config

Install as dev dep in `frontend/`:

```bash
npm install -D @playwright/test @types/node
npx playwright install --with-deps chromium
```

Generated baseline `playwright.config.ts`:

```ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',
  fullyParallel: false, // sim state is global; parallel runs interfere
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 1 : 0,
  reporter: process.env.CI ? 'github' : 'list',
  use: {
    baseURL: 'http://localhost:5173',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
  },
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
  ],
  webServer: [
    {
      command: 'cd ../backend && mvn spring-boot:run -q',
      url: 'http://localhost:8085/api/simulation/status',
      timeout: 120_000,
      reuseExistingServer: !process.env.CI,
    },
    {
      command: 'npm run dev',
      url: 'http://localhost:5173',
      timeout: 30_000,
      reuseExistingServer: !process.env.CI,
    },
  ],
});
```

**Notes:**
- `fullyParallel: false` because the simulation has global state (single tick loop). Running two specs concurrently would both hit `/api/command/start` and see each other's vehicles. Parallelism inside specs (e.g. multiple assertions in one browser context) is fine.
- `webServer.url` is what Playwright polls — for backend, `/api/simulation/status` already exists and returns 200 when the app is up.
- `reuseExistingServer` lets locally running `mvn spring-boot:run` / `npm run dev` stay up across test runs.

## 2. Existing backend endpoints suitable for stubbing

From `backend/src/main/java/.../controller/`:

| Endpoint | Method | Stubbed? | Why |
|---|---|---|---|
| `/api/simulation/status` | GET | No — always real | Used by Playwright webServer readiness check |
| `/api/maps` | GET | No | Lists default maps; real call is trivially fast |
| `/api/command` | POST | No | Test 1 actually drives sim — must be real |
| `/api/command/load-config` | POST | Can stub in test 3 | Avoids Overpass; test 1 loads via `/api/maps` flow instead |
| `/api/vision/analyze-components-bbox` | POST | **Yes** (test 2) | Avoid Claude CLI; response shape = `MapConfig` JSON |
| `/api/osm/fetch-roads` | POST | **Yes** (test 3) | Avoid Overpass API; response shape = `MapConfig` JSON |
| `/api/roads` | GET | No | Fast, served from current in-memory state |

## 5. Responsive layout assertion (mirror Phase 14)

`frontend/src/test/bugfixes.test.tsx` already tests the mobile breakpoint in Vitest (JSDOM) — the Playwright version proves it in a real browser with real CSS:

```ts
test('mobile viewport stacks layout', async ({ browser }) => {
  const ctx = await browser.newContext({ viewport: { width: 375, height: 667 } });
  const page = await ctx.newPage();
  await page.goto('/');

  const canvas = page.locator('canvas');
  const sidebar = page.getByRole('complementary'); // or a data-testid

  const canvasBox = await canvas.boundingBox();
  const sidebarBox = await sidebar.boundingBox();

  expect(canvasBox!.y + canvasBox!.height).toBeLessThanOrEqual(sidebarBox!.y);
});
```

## 6. Backend startup time constraint

`mvn spring-boot:run` cold start on this machine: ~20s first time, ~8s warm.
First CI run (no Maven cache): may hit 60s.
**Recommendation:** webServer timeout = **120_000 ms** (2 min). Local dev reuses running server anyway.

## 7. Risks and edge cases

- **Port collision:** if another service holds 8085 or 5173, webServer hangs. Mitigation: planner includes a precheck task or documents the assumption in e2e/README.
- **Maven not on PATH in CI:** would need a JDK+Maven docker image. Out of scope (no CI yet).
- **STOMP over SockJS through Playwright:** the WS library works fine in real browsers; no special handling needed. Do NOT try to stub WebSocket — tests 1, 5 need the real broker.
- **Flaky tick-based assertions:** vehicle count increments at 20 Hz. Assert "eventually >= 1" with `expect.poll(...)` rather than strict counts.

## 8. Dimension 8 (Nyquist validation) — not applicable

This phase is test infrastructure, not a physics/IDM component. No Nyquist frequency/sampling analysis needed. Validation of the phase itself is the e2e tests passing locally.

## 9. Reuse Strategy

- Do NOT replace Vitest. Vitest stays for component-level logic tests (fast, JSDOM).
- Playwright lives alongside and is invoked via a separate `npm run test:e2e` script.
- Zero changes to existing source code expected, EXCEPT adding `data-testid` attributes to a handful of elements if role-based selectors prove insufficient (planner decides per test).
