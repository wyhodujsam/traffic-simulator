# Phase 22.1: Playwright E2E test suite - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning
**Source:** User decision in closing session of Phases 21+22 (chat 2026-04-22)

<domain>
## Phase Boundary

Install and configure Playwright e2e test framework in the frontend workspace and author smoke tests for the critical user flows. Goal is to catch regressions that unit tests (Vitest) cannot — tests that require a real backend (Spring Boot) + real frontend (Vite) + real browser.

This phase is **infrastructure + smoke tests**, not comprehensive coverage. Coverage expansion is a follow-up.

</domain>

<decisions>
## Implementation Decisions

### Framework
- **`@playwright/test`** (official Playwright test runner), **NOT** `playwright/test` via Vitest bridges or custom harness. Standard, well-maintained, first-class TypeScript.
- Install in `frontend/` workspace as dev dependency. Frontend already uses npm + TypeScript so no new tooling.
- Browsers: Chromium only for MVP (fast CI, covers 90% of bugs). Firefox/WebKit can be added later if bugs warrant.

### Server orchestration
- Playwright's `webServer` config in `playwright.config.ts` must start **both** backend (Spring Boot on port 8085) and frontend (Vite on port 5173) before tests run. Two-server model because the sim pipeline is inherently full-stack (STOMP WebSocket + REST).
- Backend startup command: `cd ../backend && mvn spring-boot:run` (takes ~20s on first cold start). Timeout: 90 seconds.
- Frontend startup: `npm run dev` (fast HMR, ~3s).
- Reuse existing server if already running (`reuseExistingServer: !process.env.CI`) so local dev cycle is fast.

### Test fixtures (MVP scope)
Five smoke tests, one per critical path:

1. **Simulation start/pause** — Load "/", pick a map (e.g. `four-way-signal`), press Start, assert WebSocket connects and vehicle count increments; press Pause, assert count stops changing.
2. **AI Vision component-library flow** — Navigate to `/map`, click "AI Vision (component library)" with **a stubbed backend response** (intercept `/api/vision/analyze-components-bbox` via `page.route()`; return a canned MapConfig). Assert RoadGraphPreview renders and "Run Simulation" navigates to /.
3. **OSM bbox load** — On `/map`, pan Leaflet to a known bbox, click "Fetch roads", stub `/api/osm/fetch-roads` response, assert road count shown in sidebar + graph preview renders.
4. **Responsive layout** — 375x667 viewport (mobile): stacked layout (canvas above, controls below), sidebar not cut off. 1920x1080 viewport (desktop): side-by-side, canvas fills available width.
5. **Sim control sanity** — Load Simulation page, change speed multiplier from 1x to 3x, assert `StatsPanel` tick rate reflects change; change spawn rate, assert vehicle count ramps up.

### Stubbing policy
- Tests 2 and 3 **stub the vision + OSM endpoints** (via `page.route()` returning canned JSON) so we don't hit the Claude CLI or Overpass API in CI. The canned JSONs live in `frontend/e2e/fixtures/`.
- Test 1, 4, 5 use the **real backend + default in-memory map scenarios** (e.g. `four-way-signal.json`) — no external network.

### CI integration
- Playwright runs in its own npm script: `npm run test:e2e` (frontend workspace).
- Vitest (`npm test`) stays unchanged — different responsibilities, different speed profiles.
- Not wired into a GitHub Actions workflow in this phase (no CI config exists yet). That's a follow-up phase.

<specifics>
## Specific Ideas

- Playwright `webServer` array syntax (starts multiple servers): https://playwright.dev/docs/test-webserver
- Use `page.route('/api/vision/**', ...)` for mocking the vision endpoint; the exact URL is `/api/vision/analyze-components-bbox`.
- Reuse Phase 14 (Vitest bugfixes.test.tsx) viewport assertions as templates for the responsive spec — same breakpoints (`useIsMobile` = 768px).
- Playwright's `expect(locator).toBeVisible()` is the right assertion for "sidebar renders, not cut off".

</specifics>

<deferred>
## Deferred Ideas

- GitHub Actions CI workflow running Playwright on PRs — follow-up phase.
- Firefox + WebKit coverage — follow-up when a browser-specific bug is found.
- Visual regression snapshots — adds snapshot maintenance burden; not now.
- E2E coverage for every map scenario (`combined-loop`, `roundabout`, etc.) — MVP picks one (`four-way-signal`).
- E2E against the real Claude CLI — too slow, non-deterministic; stub in CI.
- Cross-origin / auth flows — not applicable (no auth).

</deferred>
