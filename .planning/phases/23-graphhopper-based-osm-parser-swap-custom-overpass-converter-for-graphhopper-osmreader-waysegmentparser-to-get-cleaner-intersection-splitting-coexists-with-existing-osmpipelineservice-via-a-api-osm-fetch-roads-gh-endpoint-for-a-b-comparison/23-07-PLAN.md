    ```markdown
    # OSM Converters

    The backend supports multiple OSM-to-MapConfig converters. They coexist and expose
    parallel REST endpoints so the frontend can A/B compare on the same bounding box.

    ## Active converters

    | Phase | Service                 | Endpoint                     | Parser            | Id prefix |
    |-------|-------------------------|------------------------------|-------------------|-----------|
    | 18    | `OsmPipelineService`    | `POST /api/osm/fetch-roads`    | Overpass JSON + heuristic node-reference counting | `osm-` |
    | 23    | `GraphHopperOsmService` | `POST /api/osm/fetch-roads-gh` | GraphHopper 10.2 `WaySegmentParser` (tower-node splitting) | `gh-` |
    | 24*   | `Osm2StreetsService`    | `POST /api/osm/fetch-roads-o2s` | A/B Street's osm2streets (Rust/WASM) | `o2s-` |

    (* Phase 24 not yet implemented.)

    ## Extension contract

    Every converter implements `com.trafficsimulator.service.OsmConverter`:

    ```java
    public interface OsmConverter {
        MapConfig fetchAndConvert(BboxRequest bbox);
        default String converterName() { return getClass().getSimpleName(); }
    }
    ```

    Implementations MUST:

    1. Be annotated `@Service` (Spring singleton).
    2. Accept the same `BboxRequest` DTO.
    3. Produce a `MapConfig` that passes `MapValidator` without weakening the validator.
    4. Override `converterName()` to return a stable human-readable label used in A/B logs
       (e.g. `"Overpass"`, `"GraphHopper"`, `"osm2streets"`).
    5. Throw `IllegalStateException("No roads found in selected area")` for empty bboxes
       (maps to HTTP 422 in `OsmController`).
    6. Throw `RestClientException` for upstream availability failures (maps to HTTP 503).
    7. Use a unique node-id prefix (`gh-`, `osm-`, `o2s-`, ...) to prevent collisions when
       outputs from multiple converters are compared or merged in one process.

    ## A/B fairness — the `OsmConversionUtils` contract

    To keep A/B comparisons fair, every converter MUST use `OsmConversionUtils` for the
    following shared operations. This guarantees that differences between converters come
    from OSM parsing, not from projection drift or lane-clamp rounding.

    | Concern | Constant / function |
    |---------|---------------------|
    | Canvas projection | `lonToX`, `latToY`, `CANVAS_W=1600.0`, `CANVAS_H=1200.0`, `CANVAS_PADDING=50.0` |
    | Distance | `haversineMeters` |
    | Speed defaults | `speedLimitForHighway` |
    | Lane count | `laneCountForWay` (clamp `[MIN_LANE_COUNT=1, MAX_LANE_COUNT=4]`) |
    | Bidirectional lateral offset | `laneCount * LANE_WIDTH_BACKEND (14.0) / 2.0 + 1.0` |
    | Road minimum length filter | `MIN_ROAD_LENGTH_M = 1.0` |
    | Default signal phase duration | `DEFAULT_SIGNAL_PHASE_MS = 30_000L` |
    | Signal-phase construction | `buildDefaultSignalPhases(fullNodeId, roads)` |
    | Spawn/despawn per terminal | `collectSpawnPoints`, `collectDespawnPoints` |
    | Final `MapConfig` assembly | `assembleMapConfig` |

    ## Adding a third converter (recipe)

    1. Add the parser dependency to `backend/pom.xml`.
    2. Create `MyNewConverterService implements OsmConverter` in
       `backend/src/main/java/com/trafficsimulator/service/`. Annotate `@Service`.
    3. Inside `fetchAndConvert`, call `OsmConversionUtils.assembleMapConfig(...)` with
       nodes/roads/intersections built using the shared helpers for projection, speed, etc.
    4. Add a `@PostMapping("/fetch-roads-xxx")` method to `OsmController.java` (same class,
       shared `@ExceptionHandler` taxonomy).
    5. Inject the new service via `@RequiredArgsConstructor` (add field, Lombok handles ctor).
    6. Add `MyNewConverterServiceTest` using hand-rolled OSM fixtures under
       `backend/src/test/resources/osm/`.
    7. Add 3 controller tests to `OsmControllerTest` mirroring the Phase 23 triplet
       (valid-bbox-200, empty-area-422, unavailable-503).
    8. (Optional) Extend `OsmPipelineComparisonTest` to a three-way diff.
    9. Add the matching frontend button to `MapSidebar.tsx` and handler to `MapPage.tsx`.
    10. Mirror `osm-bbox-gh.spec.ts` in `frontend/e2e/osm-bbox-xxx.spec.ts`.

    ## A/B comparison — what to measure

    When comparing two converters on the same bbox, useful metrics are:

    - Road count.
    - Intersection count (by type: SIGNAL / ROUNDABOUT / PRIORITY).
    - Node coordinate fidelity (do the same lat/lon pairs appear?).
    - Handling of unusual OSM tags (`oneway=-1`, missing `highway`, `junction=roundabout`
      without explicit `oneway=yes`).

    Invoke the A/B harness with:

    ```bash
    cd backend && ./mvnw test -Dtest=OsmPipelineComparisonTest -Dosm.online=on
    ```
