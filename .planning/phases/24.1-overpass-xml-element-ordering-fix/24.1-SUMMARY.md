# Phase 24.1: Overpass XML Element Ordering Fix — Phase Summary

**One-line query rewrite (`>;` moved into the union, single `out body qt;`) plus a four-layer regression net (unit query-shape + integration captured-body + integration service-behavior + e2e real-backend UI + live Modlin manual smoke) closes the Phase 23/24 element-ordering bug class and the test-coverage gap that hid it. Live smoke confirms all three OSM endpoints (Phase 18 Overpass, Phase 23 GraphHopper, Phase 24 osm2streets) return ≥ 1 road end-to-end on the Modlin bbox.**

## Phase contract

Phase 24.1 was inserted as a hotfix between Phase 24 (osm2streets integration) and Phase 25 (traffic flow viz) on 2026-04-25 after a manual smoke against `overpass-api.de` revealed:

- **Phase 18 (Overpass):** worked.
- **Phase 23 (GraphHopper):** 503 `Could not parse OSM file` — `WaySegmentParser.pass2` rejected the XML because `<way>` elements appeared before `<node>` elements (non-standard OSM XML).
- **Phase 24 (osm2streets):** 422 `No roads found in selected area` — `StreetNetworkMapper` couldn't resolve node coordinates because the same ordering bug short-circuited the parser.
- **Root cause:** `OverpassXmlFetcher.buildOverpassXmlQuery()` (introduced in Phase 23) emitted ways-then-nodes via the two-pass `out body; >; out skel qt;` form, instead of the standard nodes-then-ways via the single-pass `(...; >;); out body qt;` form.
- **Coverage gap that hid the bug:** every Playwright e2e stubbed the backend with `page.route`, the only real-binary integration test was `@Disabled` by default, and the canned osm2streets fixtures were OUTPUTS of the binary (bypassing the entire Overpass→XML→subprocess pipeline).

The phase fixes the bug and closes the coverage gap at four independent layers.

### Backend startup workaround needed for the smoke (pre-existing Phase 24 bug — NOT a Phase 24.1 regression)

The Phase 24 osm2streets endpoint initially returned HTTP 503 `osm2streets unavailable` when the backend was started with the default `application.properties`. Root cause:

- `application.properties` ships with `osm2streets.binary-path=backend/bin/osm2streets-cli-linux-x64` — a **project-root-relative** path.
- `mvn spring-boot:run` runs with `cwd=backend/`, so the path resolves to `backend/backend/bin/osm2streets-cli-linux-x64` — which does NOT exist — and `Osm2StreetsService.fetchAndConvert` fails on subprocess `Runtime.exec` with ENOENT.

**The smoke succeeded only after restarting the backend with an explicit absolute-path override:**

```bash
cd backend && mvn spring-boot:run -q \
  -Dspring-boot.run.jvmArguments="-Dosm2streets.binary-path=$(pwd)/bin/osm2streets-cli-linux-x64"
```

(Equivalent to Plan 02's `@TestPropertySource osm2streets.binary-path=bin/osm2streets-cli-linux-x64` workaround, which works because surefire also runs from `cwd=backend/`.)

**This is a pre-existing Phase 24 bug, NOT a Phase 24.1 regression.** It is already logged in `.planning/phases/24-osm2streets-integration-.../deferred-items.md`. The fact that it surfaced again at the Phase 24.1 manual smoke is evidence that the deferred item should be promoted: any user starting the app with `mvn spring-boot:run` (the documented dev-mode command) hits this on the first osm2streets click. **Recommendation:** open a Phase 24.2 (single-task) plan to fix `application.properties` to a cwd-resilient form (absolute `${user.dir}`-anchored, classpath-resolved, or `@Value`-derived from a known anchor). Add a backend startup integration check (Spring `ApplicationRunner` or `@PostConstruct` health check) that verifies the binary path resolves before the first request.

## Key decisions (phase-wide)

| Decision                                                                                | Choice                                                                  | Rationale                                                                                                                                                                                                                                                       |
| --------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Query form (Plan 01)                                                                    | `(way[...](...);>;);out body qt;` (recurse in union, single `out`)      | Produces standard OSM XML (nodes-before-ways). Matches what osm2streets and most OSM tooling expect. Verified by direct curl + downstream consumer ingestion.                                                                                                  |
| `out body` vs `out skel` (Plan 01)                                                      | `out body qt;`                                                           | `body` keeps full node tags (e.g. `highway=traffic_signals`). `skel` would strip them — Phase 24 lane-level metadata depends on `body`.                                                                                                                          |
| Integration test gating (Plan 02)                                                       | Default-enabled (`@EnabledOnOs(LINUX)` only)                            | Whole point of the phase is closing the test gap. Disabled-by-default would re-create it. Linux-only because the bundled binary is linux-x64 and CI is Linux.                                                                                                  |
| Bean wiring for `MockRestServiceServer` (Plan 02)                                       | Singleton `RestClient.Builder` + ordered bean methods                   | Spring 6.1's `MockRestServiceServer.bindTo(...)` only accepts `RestClient.Builder` (not built `RestClient`); auto-configured builder is prototype-scoped. Custom singleton is required to ensure binding and building share the same builder instance.        |
| Provenance comment placement in fixture XML (Plan 02)                                   | INSIDE the `<osm>` root element                                          | GraphHopper's `OSMInputFile.openXMLStream` calls `XMLStreamReader.next()` once and expects the first event to be `START_ELEMENT` "osm" — a leading XML comment trips `IllegalArgumentException`. osm2streets-cli accepts both placements.                       |
| Spring property override syntax for Playwright webServer (Plan 03)                      | `-Dspring-boot.run.jvmArguments=-D...` (NOT `-Dspring-boot.run.arguments=--...`) | `arguments` is comma-separated and corrupts multi-value overrides (the second `--key=value` ends up appended to the first key's value). `jvmArguments` is space-separated and passes each `-D` to the JVM as an independent system property.                |
| Fixture-server stack (Plan 03)                                                          | Pure Node stdlib (`node:http`, `node:fs`, `node:path`)                  | Zero npm dependencies. Drops into Playwright's `webServer` block via `node ./e2e/server/overpass-fixture-server.cjs` directly. No Express, no nodemon, no transpile.                                                                                            |
| Real-backend spec count assertion (Plan 03)                                             | Regex `^([1-9]\d*) roads?, \d+ intersections?$` (≥ 1 road)              | Real-backend output from StreetNetworkMapper may collapse the canned XML's 2 ways differently than the stubbed spec's pre-canned `3 roads, 2 intersections` JSON. Pin "≥ 1", not exact count.                                                                  |
