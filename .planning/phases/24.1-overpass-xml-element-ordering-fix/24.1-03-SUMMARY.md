### Backend startup workaround (pre-existing Phase 24 binary-path bug)

Phase 24 osm2streets initially returned HTTP 503 `osm2streets unavailable` with the default config because of the same binary-path resolution bug Plan 02 documents in its Deviation 3:

- `application.properties` ships with `osm2streets.binary-path=backend/bin/osm2streets-cli-linux-x64` (a project-root-relative path).
- `mvn spring-boot:run` runs with `cwd=backend/`, so the path resolves to `backend/backend/bin/osm2streets-cli-linux-x64` — which doesn't exist — and `Osm2StreetsService` throws on subprocess `Runtime.exec`.

The smoke succeeded only after restarting the backend with `-Dspring-boot.run.jvmArguments="-Dosm2streets.binary-path=$(pwd)/backend/bin/osm2streets-cli-linux-x64"` (absolute path). This is the SAME pre-existing bug logged in **Phase 24's `deferred-items.md`** (binary-path resolution) — **NOT in scope for Phase 24.1**, which only fixes the Overpass query-shape issue.

**Why this matters for the SUMMARY:** the user-reported failure mode at the live smoke is a known Phase 24 bug, not a Phase 24.1 regression. The Phase 24.1 fix (Plan 01 query-shape rewrite) is correct and verified. The follow-up to fix the binary-path resolution is its own item:

- **Recommendation:** expand Phase 24's `deferred-items.md` entry to flag this as user-reported during the Phase 24.1 manual smoke (raises priority), OR open a Phase 24.2 to fix `application.properties` to use a path that resolves correctly under both `mvn spring-boot:run` (cwd=`backend/`) and Spring Boot fat-JAR runs (cwd may be elsewhere). Reasonable fixes: classpath-resolved location, absolute `${user.dir}/bin/...`, or a `@Value`-injected default that derives an absolute path from a known-good anchor.
- **Plan 02 already documents the same workaround** for the integration test (`@TestPropertySource osm2streets.binary-path=bin/osm2streets-cli-linux-x64`) — this strengthens the case that the production default needs fixing.

## Decisions Made

- **Decision (Task 2-Decision):** PROCEED. Spike confirmed override mechanism works on first try, well inside budget. Tasks 3-5 implemented in full.
- **Override form (FINAL — discovered during Task 5 deviation, after the spike):** `-Dspring-boot.run.jvmArguments=-D...` instead of `-Dspring-boot.run.arguments=--...`. The `arguments` form is comma-separated and corrupts multi-value overrides; the `jvmArguments` form is space-separated and passes each `-D` to the JVM cleanly.
- **Fixture-server stack:** pure Node stdlib (zero npm deps). The Playwright webServer block runs `node ./e2e/server/overpass-fixture-server.cjs` directly — no Express, no nodemon, no transpile. Total dependency surface = `node:http`, `node:fs`, `node:path`.
- **Bbox-aware fixture is NOT required.** The fixture server replies to ANY POST `/api/interpreter` with the same canned XML. Plan 01's unit test pins the query SHAPE; Plan 02's `OsmPipelineSmokeIT.regressionGuard` pins the captured request BODY shape. The Plan 03 spec exists to catch FRONTEND-side regressions on real-backend output, not query-shape regressions.
- **Counts loosened to `\d+ roads?, \d+ intersections?` regex** — real-backend output from StreetNetworkMapper may collapse the canned XML's 2 ways differently than the stubbed spec's pre-canned `3 roads, 2 intersections` JSON; we assert ≥ 1 road, not an exact count.
- **API-then-UI smoke pattern:** `request.post(/api/osm/fetch-roads-o2s)` runs INSIDE the same test, BEFORE the UI assertions. If the backend regresses, the test fails fast with the exact body — much faster triage than the UI-only assertion path.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] [Rule 3 — Blocking] `-Dspring-boot.run.arguments=--osm.overpass.urls=...` corrupts the second override**

- **Found during:** Task 5 RED iteration (commit `7e273d2` body documents the failure mode in detail).
- **Issue:** The Task 4 plan's working override form from the Task 1 spike (`-Dspring-boot.run.arguments=--osm.overpass.urls=http://localhost:18086`) worked when there was only ONE override. As soon as Task 5 needed BOTH `--osm.overpass.urls=...` AND `--osm2streets.binary-path=bin/osm2streets-cli-linux-x64`, the spring-boot-maven-plugin parsed the comma-separated `arguments` value and produced `mirrors=[http://localhost:18086, --osm2streets.binary-path=bin/osm2streets-cli-linux-x64]` — the URL list now contained the binary-path argument, which the second `--` was meant to set as a separate property. Symptom: backend log showed `Overpass attempt: --osm2streets.binary-path=bin/osm2streets-cli-linux-x64/api/interpreter` and the test failed with a connection-refused error against an obviously corrupt URL.
- **Fix:** Switched to `-Dspring-boot.run.jvmArguments=-Dosm.overpass.urls=http://localhost:18086 -Dosm2streets.binary-path=bin/osm2streets-cli-linux-x64`. `jvmArguments` is space-separated (not comma-separated) and passes each `-D` directly to the JVM as an independent system property, which Spring's `@Value` reads cleanly. Both overrides applied correctly; backend log now shows `mirrors=[http://localhost:18086]` (correct) AND `Loaded osm2streets-cli at bin/osm2streets-cli-linux-x64` (correct).
- **Files modified:** `frontend/playwright.config.ts` (the `command:` string in the backend webServer entry).
- **Verification:** `osm-bbox-real-backend` test now passes 1/1 in 14.4s; full e2e suite 9/9 in 30.1s.
- **Commit:** `625715a` (`fix(24.1-03): switch backend webServer override to jvmArguments form (Rule 1+3 deviation)`).

### Out-of-scope discoveries (not auto-fixed; logged elsewhere)

**2. Pre-existing Phase 24 osm2streets binary-path resolution bug (re-surfaced at the live manual smoke)**

- **Found during:** Task 6 manual smoke. Backend started with default config returned HTTP 503 `osm2streets unavailable` because `osm2streets.binary-path=backend/bin/osm2streets-cli-linux-x64` is repo-root-relative but `mvn spring-boot:run` runs with `cwd=backend/`.
