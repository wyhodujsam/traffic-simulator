# Phase 24.1 Plan 03 — Spike notes (Task 1)

## Question

Does Spring's `mvn spring-boot:run` reliably propagate a command-line override of the
`osm.overpass.urls` property to `OverpassXmlFetcher`'s `@Value` injection, so a Playwright
`webServer` block can redirect the backend's outbound Overpass calls to a local fixture HTTP
server (`http://localhost:18086`) instead of `overpass-api.de`?

## Working command

```bash
cd backend && mvn spring-boot:run -q \
  -Dspring-boot.run.arguments=--osm.overpass.urls=http://localhost:18086
```

This is the form that worked on first try. The alternate
`-Dspring-boot.run.jvmArguments="-Dosm.overpass.urls=..."` form was NOT tried because the first
form already met the acceptance criterion.

## Log evidence (deterministic)

After starting the backend with the working command above and triggering one Overpass-bound
call:

```bash
curl -sS -X POST http://localhost:8086/api/osm/fetch-roads-o2s \
  -H 'Content-Type: application/json' \
  -d '{"south":52.431,"west":20.65,"north":52.438,"east":20.662}'
```

The backend log shows the override took effect — the configured mirrors list AND the actual
HTTP attempt line both point at `localhost:18086`, NOT `overpass-api.de`:

```
2026-04-25T18:59:52.126+02:00  INFO  c.t.service.OverpassXmlFetcher
  : Fetching OSM data (XML) for bbox: BboxRequest[south=52.431, west=20.65, north=52.438, east=20.662]
    (mirrors=[http://localhost:18086])

2026-04-25T18:59:52.131+02:00  INFO  c.t.service.OverpassXmlFetcher
  : Overpass attempt: http://localhost:18086/api/interpreter

2026-04-25T18:59:52.165+02:00  WARN  c.t.service.OverpassXmlFetcher
  : Overpass mirror http://localhost:18086 failed: I/O error on POST request for
    "http://localhost:18086/api/interpreter": Connection refused
```

The `Connection refused` is EXPECTED at the spike stage — no fixture server is running on
`:18086` yet. The load-bearing observation is that the backend is hitting the override URL
(not `overpass-api.de`), which proves the Spring property override pathway works.

The endpoint returned `HTTP 503 — "Overpass API unavailable"` to the curl client, exactly as
expected when all configured mirrors fail. That failure mode is the contract for an empty/down
mirror list — once the local fixture server is brought up in Plan 03 Task 3 it will return 200
with canned XML and the pipeline will succeed.

## Decision

**OUTCOME: PROCEED.** The Spring property override works on the first attempt, well within the
30-minute spike budget. Plan 03 Tasks 3-5 are unblocked and can use the working command
verbatim in `frontend/playwright.config.ts`'s second `webServer` entry.

## Elapsed

Wall-clock spike duration: **~36 seconds** (well under the 30-minute target and the 60-minute
hard cap). The fast turnaround was helped by:

- Backend startup time on this machine is ~9 seconds (warm Maven local cache, JIT-warm OpenJDK
  17.0.18).
- The override syntax found on first try — no fallback to the `jvmArguments` form needed.
- No spurious Java/Spring exception during startup with the override; the property is read
  cleanly by `OverpassXmlFetcher`'s `@Value("${osm.overpass.urls:https://overpass-api.de}")`.

## Cleanup

Backend process killed at end of spike (`pkill -f 'spring-boot:run'`). Port 8086 confirmed
free via `ss -tlnp | grep ':8086 '` returning no rows. The user will start their dev backend
back up at the Task 6 manual smoke checkpoint.
