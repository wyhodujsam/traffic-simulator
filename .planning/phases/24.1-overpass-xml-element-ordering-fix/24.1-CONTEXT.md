# Phase 24.1 — Overpass XML Element Ordering Fix

## Bug summary

`OverpassXmlFetcher.buildOverpassXmlQuery()` (introduced in Phase 23, shared by Phase 24) builds an Overpass query whose XML output emits `<way>` elements **before** `<node>` elements:

```overpassql
[out:xml][timeout:25];
(way["highway"~"^(motorway|trunk|primary|secondary|tertiary|unclassified|residential|living_street)$"](S,W,N,E););
out body;     -- emits ways
>;            -- recurses to find referenced nodes
out body qt;  -- emits nodes (AFTER ways)
```

Standard OSM XML format requires nodes to appear before ways. With ways first:
- **osm2streets** (Phase 24): silently returns 0 roads, 0 intersections — `parseRoads` finds the tuple-array but downstream `StreetNetworkMapper` cannot resolve node coordinates → `IllegalStateException("No roads found in selected area")` → HTTP 422.
- **GraphHopper** (Phase 23): throws "Could not parse OSM file" — `WaySegmentParser.pass2` fails when reading nodes after ways → HTTP 503 from `OsmController` unexpected-error handler.
- **Phase 18** (`OsmPipelineService`): unaffected — uses its own fetcher that does not go through `OverpassXmlFetcher`.

Verified locally on bbox `(52.431, 20.65, 52.438, 20.662)` — Modlin / Nowy Dwór Mazowiecki area:
- Phase 18: 17 nodes, 18 roads, 3 intersections — works ✓
- Phase 23: 503 "Could not parse OSM file" ✗
- Phase 24: 422 "No roads found in selected area" ✗
- `osm2streets-cli` binary fed the SAME bbox via direct curl `(way["highway"](...);>;);out;` returns 161 roads / 132 intersections — confirms the binary is fine, the bug is upstream.

## Coverage gap that hid this

1. **Playwright e2e** (`osm-bbox-o2s.spec.ts`, `osm-bbox-gh.spec.ts`): both **stub** the backend endpoint with canned JSON via `page.route('**/api/osm/fetch-roads-*', ...)`. Real backend, real Overpass, real binary never exercised.
2. **`OsmPipelineComparisonTest`** (the only end-to-end real-binary harness): `@Disabled` by default, only runs with `-Dosm.online=on` to avoid hammering Overpass in CI.
3. **Canned osm2streets fixtures** (`backend/src/test/resources/osm2streets/*.json`): these are *outputs* of the binary — they bypass the entire Overpass→XML→subprocess pipeline. The mapper tests against fixtures are green, hiding the upstream bug.
4. **No integration smoke** with mocked Overpass + real binary subprocess. That is exactly the gap this phase fills.

## Scope

### In scope
1. **Fix** `OverpassXmlFetcher.buildOverpassXmlQuery()` to emit nodes-before-ways (single-pass union recursion + single `out body qt;`).
2. **Unit test** asserting the produced query string is correct (regex / structural assertion).
3. **Backend integration test** — new `OsmPipelineSmokeIT` (or extend existing) using `MockRestServiceServer` / WireMock to serve a canned Overpass XML response, then run the **real** `Osm2StreetsService.fetchAndConvert()` (via subprocess) and `GraphHopperOsmService.fetchAndConvert()`. Asserts both return ≥ 1 road. **NOT** disabled by default — runs in `mvn test`.
4. **Playwright e2e** that exercises the real backend through one OSM endpoint (osm2streets and/or GraphHopper) without stubbing the API. Backend's Overpass URL is overridden to a Playwright-served local URL (Playwright `webServer` augmented or test fixture HTTP server) so CI doesn't depend on Overpass.

### Out of scope
- Refactoring `OverpassXmlFetcher` beyond the query fix.
- Touching Phase 18 (`OsmPipelineService`) — it has its own working fetcher, not the shared helper.
- Adding lane-aware rendering for osm2streets `lanes[]` metadata (deferred, future work).
- The pre-existing `backend/target/` tracking issue (deferred from Phase 24).

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Query form | `(way[...](...);>;);out body qt;` (recurse in union, single `out`) | Produces standard OSM XML (nodes-before-ways). Matches what osm2streets and most OSM tooling expect. |
| Integration test runs by default | YES | The whole point of this phase is closing the test gap. Disabled-by-default would re-create the gap. Use `MockRestServiceServer` so no network. |
| Playwright real-backend path | Override `osm.overpass.mirrors` via Spring property to local URL Playwright handles | Avoids hitting Overpass in CI; exercises real backend code path. |
| Phase numbering | 24.1 (inserted between 24 and 25) | Follows Phase 22.1 precedent for inserted hotfix phases. Both Phase 23 and Phase 24 affected, but the fix is in code that landed with Phase 23 and that became user-visible at Phase 24 merge. |
| Branch | `feat/phase-24.1-overpass-xml-ordering` | Standard convention. |

## Acceptance criteria

1. `OverpassXmlFetcher.buildOverpassXmlQuery()` produces a query whose Overpass XML output puts `<node>` elements before `<way>` elements.
2. New unit test in `OverpassXmlFetcherTest` asserts the query format constraint and would fail on the pre-fix string.
3. New backend integration test serves a canned Overpass XML response and validates that:
   - `Osm2StreetsService.fetchAndConvert(bbox)` returns a `MapConfig` with ≥ 1 road and passes `MapValidator`.
   - `GraphHopperOsmService.fetchAndConvert(bbox)` returns a `MapConfig` with ≥ 1 road and passes `MapValidator`.
   - Test runs in default `mvn test` (NOT `@Disabled`).
4. New Playwright spec hits the real `/api/osm/fetch-roads-o2s` endpoint (no stub) against a backend whose Overpass mirror is redirected to a local fixture; asserts the sidebar shows ≥ 1 road in the result panel.
5. Manual smoke: live Modlin bbox returns ≥ 1 road via osm2streets and via GraphHopper through the running app at `http://localhost:5173/map`.
6. All existing tests (308+ backend, 59 frontend unit, 8 frontend e2e) remain green — no regressions.

## Risks and unknowns

- **Playwright URL override:** Playwright cannot intercept backend HTTP calls. Need a backend-side mechanism (Spring profile, env var, system property) to redirect `osm.overpass.mirrors`. Risk: clean test plumbing requires a small refactor or a `@TestConfiguration`. Mitigation: if the Playwright path turns out to need >2 hours of plumbing, defer the e2e and rely on the backend integration test, document this in the SUMMARY.
- **Overpass `qt` ordering semantics:** `out body qt;` in a single union may not guarantee nodes-first if the union mixes types. Need to verify the produced XML against the actual Overpass server before claiming the fix.
- **Canned Overpass fixture freshness:** the fixture used in integration tests becomes stale as OSM evolves. Mitigation: pin the fixture to a small, well-known geometry (3-way intersection, 1 highway) and freeze it; document where it came from.
