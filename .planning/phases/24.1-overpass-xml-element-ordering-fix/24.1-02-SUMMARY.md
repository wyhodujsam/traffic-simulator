## Decisions Made

- **Bean-ordering wiring** for `MockRestServiceServer`: Spring 6.1's `MockRestServiceServer.bindTo(...)` only accepts `RestClient.Builder` (not a built `RestClient`); auto-configured `RestClient.Builder` is prototype-scoped. We therefore declared our own singleton `@Bean RestClient.Builder overpassTestRestClientBuilder()` and built BOTH the `MockRestServiceServer` and the `@Primary RestClient` from it, with bean-creation ordering enforced by listing the mock-server bean as a method parameter on the RestClient bean factory.
- **`spring.main.allow-bean-definition-overriding=true`** at `@TestPropertySource` level (Spring Boot 2.1+ disables override by default; we opt in for this `@SpringBootTest` only тАФ does NOT affect production beans).
- **Provenance comment placement** inside `<osm>` root, not before it: `GraphHopper.OSMInputFile.openXMLStream` calls `XMLStreamReader.next()` once and expects the first event to be `START_ELEMENT` "osm". A leading XML comment becomes a `COMMENT` event and trips the `IllegalArgumentException("File is not a valid OSM stream")`. Standard XML-comment placement is preserved (still inside the document, still part of the OSM contract); osm2streets-cli has no problem with it either.
- **`osm2streets.binary-path=bin/osm2streets-cli-linux-x64`** (cwd-relative): mvn surefire runs from `backend/`, but the production-default path `backend/bin/...` is rooted at the repo root. Test override is the smallest possible change; production behaviour is unaffected.
- **`@EnabledOnOs(OS.LINUX)`** as the only gate: bundled binary is linux-x64 only. Acceptable because (a) CI is Linux (b) the only alternative is a hard system-property gate, which would re-create the gap CONTEXT.md identified.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 тАФ Blocking] `MockRestServiceServer.bindTo(RestClient)` does not exist in Spring 6.1**

- **Found during:** Task 2 RED-iteration 1 тАФ initial test failed with `UnknownHostException: test-overpass.local` because the auto-configured `RestClient.Builder` is prototype-scoped, so the builder we bound the mock to was a different instance from the one used to build the `RestClient` injected into `OverpassXmlFetcher`.
- **Fix:** Declared a singleton `@Bean RestClient.Builder overpassTestRestClientBuilder()` in the inner `@TestConfiguration` and used it for BOTH the `MockRestServiceServer.bindTo(...)` call AND the `@Primary RestClient` bean factory. Bean-creation ordering enforced by listing the mock-server bean as a parameter of the RestClient method.
- **Files modified:** `backend/src/test/java/com/trafficsimulator/service/OsmPipelineSmokeIT.java` only (test-side wiring; no production code touched).
- **Verification:** all 3 OsmPipelineSmokeIT tests now PASS.
- **Plan-doc impact:** plan suggested the workaround in its `<interfaces>` block ("if `bindTo(RestClient)` does not work, register a `@TestConfiguration` exposing a builder"); we landed exactly that workaround.

**2. [Rule 3 тАФ Blocking] `BeanDefinitionOverrideException` on the `overpassRestClient` bean**

- **Found during:** Task 2 RED-iteration 0 тАФ Spring Boot 2.1+ disables bean-definition override by default; our `@TestConfig.@Primary RestClient overpassRestClient(...)` could not override `OsmClientConfig.overpassRestClient`.
- **Fix:** Added `spring.main.allow-bean-definition-overriding=true` to the test's `@TestPropertySource` properties array. Test-only knob; production config unchanged.
- **Files modified:** `backend/src/test/java/com/trafficsimulator/service/OsmPipelineSmokeIT.java` only.
- **Verification:** ApplicationContext now loads cleanly.

**3. [Rule 3 тАФ Blocking] osm2streets-cli binary path is repo-root-relative**

- **Found during:** Task 2 RED-iteration 2 тАФ `Osm2StreetsCliException: Cannot run program "backend/bin/osm2streets-cli-linux-x64": error=2, No such file or directory`. Mvn surefire runs from `backend/`, so the production default path resolves to `backend/backend/bin/...` which doesn't exist.
- **Fix:** Added `osm2streets.binary-path=bin/osm2streets-cli-linux-x64` to the test's `@TestPropertySource` properties тАФ cwd-relative so it works from the surefire JVM cwd. Production default unchanged.
- **Files modified:** `backend/src/test/java/com/trafficsimulator/service/OsmPipelineSmokeIT.java` only.
- **Verification:** Osm2StreetsService.fetchAndConvert now executes the binary; test passes.

**4. [Rule 3 тАФ Blocking] GraphHopper rejects fixture XML (leading comment before `<osm>`)**

- **Found during:** Task 2 RED-iteration 2 тАФ `IllegalArgumentException: File is not a valid OSM stream` from `OSMInputFile.openXMLStream` when GraphHopper's `WaySegmentParser` opened the temp `.osm` file produced by `OverpassXmlFetcher.fetchXmlToTempFile`. Bytecode inspection of `OSMInputFile` showed that `openXMLStream` calls `XMLStreamReader.next()` exactly once and expects the first event to be `START_ELEMENT` "osm" тАФ a leading XML comment fails that check.
- **Fix:** Moved the multi-line provenance comment in `backend/src/test/resources/osm/overpass-real-modlin.xml` from BEFORE the `<osm>` root element to INSIDE it (immediately after the opening `<osm ...>` tag). Comment text preserved (and expanded with the GraphHopper rationale); `grep -q 'Phase 24.1 fixture'` still succeeds.
- **Files modified:** `backend/src/test/resources/osm/overpass-real-modlin.xml`.
- **Verification:** GraphHopper now parses the fixture cleanly; test passes. osm2streets-cli also accepts the new placement (no regression for the other consumer).
